## 1. Ablation B：RPCA 原型设计消融

### 1.1 这个消融在证明什么

**核心问题：** RPCA 里"可学习原型库"这个设计本身有没有必要？它学到了真实的 RGB 语义，还是只是在记忆训练集的均值？它是否真正替代了"推理时依赖实时 RGB"？

**论文价值：** Ablation B 是本工作最强的创新性证明。如果可学习原型 ≥ 均值原型，说明它学到了结构化的 RGB 分布而非简单平均；如果和双流实时接近但推理不依赖 RGB，就证明了"零依赖替代双流"的意义。

### 1.2 消融表结构

| 行号 | RPCA 变体 | K/V 来源 | 推理需 RGB？ | mAP@0.5 |
|:---:|:---|:---|:---:|:---:|
| B-1 | 双流实时（接近 M-SpecGene） | 训练时：实时 RGB 特征 | ✓ | - |
| B-2 | 均值原型（Frozen Mean） | 离线计算训练集均值，冻结 | ✗ | - |
| **B-3** | **可学习原型库（ours, K=512）** | 训练时反传，推理时冻结 | **✗** | **主实验** |

### 1.3 B-2：均值原型（Frozen Mean Prototype）

#### 在做什么

用训练集所有 RGB 图像经过冻结 DINO backbone 提取特征后，计算 **全局特征均值向量 [256]**，将其广播复制为 **K=512 个相同的向量** 作为固定原型，注册为模型的 `buffer`（不参与梯度更新）。

RPCA 的交叉注意力结构不变（Q 仍然来自红外特征），只是 K/V 从"可学习 Parameter"换成了"离线计算的固定 Buffer"。

#### 意义

若 B-3（可学习原型）> B-2（均值原型），证明原型库不是简单记忆了一个平均值，而是在学习结构化的 RGB 特征分布。这是回应"原型库有没有实质上学到东西"这个质疑的关键对照。

若 B-3 ≈ B-2，则说明可学习原型的训练可以简化为一次性离线均值计算，需要重新审视 RPCA 设计。

#### 需要改动的文件和位置

**1. 新建 `src/csma_variants.py`**  
在其中定义 `FrozenMeanRPCA` 类（参考 `csma.py` 的 `RGBPrototypeCrossAttention`）：
- 构造函数接收一个预计算的 `mean_proto` 张量 `[256]`
- 将 `mean_proto.expand(K, -1)` 注册为 `self.register_buffer("rgb_prototypes", ...)`，而非 `nn.Parameter`
- `forward()` 逻辑与原版 RPCA 完全相同（cross-attention + FFN + LayerNorm）

在同文件中定义 `CSMAMeanProto` 类（参考 `csma.py` 的 `CSMA`）：
- 将 `self.rpca` 换为 `FrozenMeanRPCA`
- 其余（IRE、PD）与原版 `CSMA` 完全相同

**2. 新建均值原型预计算脚本（一次性运行）**  
建议在 `src/` 下新建 `compute_mean_proto.py`，逻辑如下：
- 加载 `FlirV1PairedDataset`（仅使用 `rgb_pixel_values`）
- `no_grad` 遍历全部训练集，调用 `extract_dino_backbone_features` 获取 `F_rgb [B, L, 256]`
- 对所有样本的所有 patch 维度求均值，得到 `mean_proto [256]`
- 保存为 `outputs_csma/mean_proto.pt`

**3. 修改 `src/train_csma.py`（或新建 `train_ablation_b.py`）**  
加载主实验同款 `CSMAConfig`，但实例化时用 `CSMAMeanProto` 替代 `CSMA`。训练逻辑（loss、optimizer、GMM-CMSS）完全不变，只是 RPCA 内部的 K/V 来源不同。

#### 运行顺序

```
第一步：python -m src.compute_mean_proto \
           --data-root FLIR_License/train \
           --out outputs_csma/mean_proto.pt

第二步：python -m src.train_csma \
           --variant mean_proto \
           --mean-proto-path outputs_csma/mean_proto.pt \
           --dataset flir_v1 \
           --data-root FLIR_License/train \
           --out-dir outputs_abl_b2_mean_proto \
           --epochs 100
```

### 1.4 B-1：双流实时（讨论）

#### 实现难点分析

B-1（双流实时）是 M-SpecGene 的做法：**训练时** RPCA 的 K/V 来自当前 batch 的真实 RGB 图像经过 DINO backbone 的实时特征，**不是可学习参数**。

这带来一个根本矛盾：

| 阶段 | 双流实时的 K/V 来源 | 问题 |
|:---|:---|:---|
| 训练时 | 当前 batch 的 `F_rgb`（实时计算） | 可行，配对数据有 RGB |
| 推理时 | **没有 RGB 图像** | 无法运行，需要 fallback |

推理时的两种 fallback 方案：

**方案 A：**  回退到"测试集 RGB 均值"作为 K/V → 本质上和 B-2（均值原型）退化为同一件事，失去独立性。

**方案 B：**  用训练集 RGB 特征的聚类中心（K-means K=512）作为推理时的固定 K/V → 实现复杂，且和 B-3 差别仅在"离线聚类"vs"端到端反传"，可作为更细粒度的对照。

#### 建议

B-1 的实现代价高（需要改 RPCA forward 接受动态 K/V 输入，且推理方案不清晰），且论文价值主要在于证明 B-3 可以替代双流，而非 B-1 本身的性能多好。

**推荐方案：** 在论文中说明"双流实时推理时需要 RGB，不满足即插即用要求"，将 B-1 作为"理论上界"处理——引用 M-SpecGene 在 FLIR 上的公开数字填入该行，而非重新实现训练。这样可以省去大量实现工作，且更有说服力。

如果审稿人坚持要求复现 B-1，再按方案 B 实现。

## 2. Ablation C：GMM-CMSS 策略消融

### 2.1 这个消融在证明什么

**核心问题：** 三阶段渐进课程里，哪一层设计真正有效？

```
Row C-1 随机掩码            ← "L_align 本身有没有用？"
  ↓ 加入 CMSS 感知
Row C-2 固定 CMSS 阈值      ← "选择性对齐目标区域有效吗？"
  ↓ 加入 GMM 动态估计
Row C-3 GMM 动态 + 单阶段B  ← "GMM 动态阈值比固定更好吗？"
  ↓ 加入三阶段课程 + 动态 λ
Row C-4 完整方案 A→B→C      ← 主实验，预期最高
```

每行和上一行**只差一个变量**，mAP 的每次增长能归因于对应设计。

### 2.2 消融表结构

| 行号 | 配置描述 | λ 策略 | 掩码策略 | 是否需要 GMM | mAP@0.5 |
|:---:|:---|:---:|:---:|:---:|:---:|
| C-1 | 随机掩码 | 固定 0.5/0.5 | 随机（按 mask_ratio=0.75） | ✗ | - |
| C-2 | 固定 CMSS 阈值 μ₂=0.5 | 固定 0.5/0.5 | CMSS > 0.5 掩蔽背景 | ✗ | - |
| C-3 | GMM-CMSS 单阶段 B | 固定 0.5/0.5 | GMM 概率采样 | ✓ | - |
| **C-4** | **三阶段渐进 A→B→C** | **动态** | **GMM** | **✓** | **主实验** |

### 2.3 需要改动的文件（总览）

所有 C 系列消融**只需改两个地方**：

1. **`src/config.py`** — 新增字段 `cmss_ablation_mode: str = "none"`
2. **`src/train_csma.py`** — 在 L_align 计算段加分支，加 CLI 参数

CMSS 计算函数本身（`cmss_utils.py`）**不需要修改**，直接复用现有 `compute_cmss`、`build_cmss_mask`。

### 2.4 C-1：随机掩码 + 固定 λ

#### 在做什么

L_align 正常计算，但完全**不使用 CMSS**，不区分目标和背景。直接按 `mask_ratio=0.75` 每次随机挑 25% 的 patch 参与对齐。λ_align 和 λ_det 全程固定为 0.5，不随 epoch 变化。

CMSS 计算（`compute_cmss`）跳过，GMM 更新跳过。

#### 涉及代码位置

**`src/train_csma.py`**，L_align 计算块（约 730–748 行）：

> **原逻辑：**  
> `compute_cmss(feat_rgb, feat_ir)` → `build_cmss_mask(stage, μ₁/₂/₃)` → `compute_align_loss`

> **C-1 改为：**  
> 跳过 compute_cmss，直接用 `torch.rand(B, L) < mask_ratio` 生成二值掩码 → `compute_align_loss`

**`src/train_csma.py`**，epoch 循环顶部，损失权重获取：

> **原逻辑：**  
> `lambda_align, lambda_det = cmss_sched.get_loss_weights(epoch)`（动态）

> **C-1 改为：**  
> `lambda_align, lambda_det = 0.5, 0.5`（固定）

**`src/train_csma.py`**，GMM 更新触发处（约 651–662 行）：

> **C-1 改为：** 跳过 GMM 更新（`gmm_update_every` 有效地设为 `total_epochs + 1`，或直接在分支里跳过）

**`src/config.py`**，新增：

> `cmss_ablation_mode: str = "none"` （合法值：`"none"` / `"random_mask"` / `"fixed_threshold"` / `"gmm_single_b"`）

#### 运行命令

```bash
python -m src.train_csma \
    --dataset flir_v1 \
    --data-root FLIR_License/train \
    --out-dir outputs_abl_c1_random_mask \
    --cmss-ablation-mode random_mask \
    --epochs 100
```

### 2.5 C-2：固定 CMSS 阈值 + 固定 λ

#### 在做什么

引入 CMSS，但阈值 μ₂ 固定为 **0.5**（不随 GMM 动态更新）。  
掩码逻辑始终使用 Stage A 的规则：`mask = (cmss_map > 0.5).float()`，即掩蔽高 CMSS 的背景 patch，让 L_align 专注于低 CMSS 的目标区域。λ 全程固定 0.5/0.5，不切换阶段。

#### 和 C-1 的唯一差别

C-1 随机选 25% patch；C-2 按 CMSS 值判断哪些 patch 是"目标"并选择性保留。若 C-2 > C-1，说明 **CMSS 感知的选择性对齐有效**。

#### 涉及代码位置

**`src/train_csma.py`**，L_align 计算块：

> **C-2 改为：**  
> 正常调用 `compute_cmss(feat_rgb, feat_ir)` 得到 `cmss_map [B, L]`  
> 调用 `build_cmss_mask(cmss_map, stage=0, mu1=0.2, mu2=0.5, mu3=0.8, gmm=None)`（`stage` 硬编码为 0，μ 硬编码为固定值，gmm=None）  
> 调用 `compute_align_loss`

**GMM 更新：** 跳过（不需要 GMM，不计算 sorted_means）

**损失权重：** 与 C-1 一样固定为 0.5/0.5

#### 关于固定 μ₂=0.5 的合理性

0.5 是初始化默认值（见 `cmss_utils.py` 第 20 行的 `_DEFAULT_SORTED_MEANS = (0.2, 0.5, 0.8)`），表示"完全不依赖 GMM 动态估计，始终用统计先验"。这是和 C-3（GMM 动态 μ₂）对比的关键。

#### 运行命令

```bash
python -m src.train_csma \
    --dataset flir_v1 \
    --data-root FLIR_License/train \
    --out-dir outputs_abl_c2_fixed_threshold \
    --cmss-ablation-mode fixed_threshold \
    --epochs 100
```

### 2.6 C-3：GMM-CMSS 单阶段 B + 固定 λ

#### 在做什么

GMM 正常每 10 epoch 更新一次，μ₁/μ₂/μ₃ 动态演化。  
掩码逻辑始终使用 **Stage B**：从拟合好的 GMM 分布中采样随机数，按排序保留 25% 的 patch（GMM 加权随机采样）。全程固定 λ=0.5/0.5，**不切换到 Stage A 或 Stage C**。

#### Stage B 掩码的具体逻辑（不需要新写，复用 `build_cmss_mask(stage=1)`）

从 GMM 采样 B×L 个值，排序后选最小的 25%（对应 GMM 分布最"低 CMSS"端的 patch），标记为 mask=0（保留），其余为 mask=1（掩蔽）。

#### 和 C-2 的唯一差别

C-2 用固定阈值 μ₂=0.5；C-3 用 GMM 动态估计阈值，且用概率采样而非硬阈值分割。若 C-3 > C-2，说明 **GMM 动态估计分布比固定阈值有效**。

#### 涉及代码位置

**`src/train_csma.py`**，L_align 计算块：

> **C-3 改为：**  
> 正常调用 `compute_cmss`  
> 调用 `build_cmss_mask(cmss_map, stage=1, mu1=mu1, mu2=mu2, mu3=mu3, gmm=cmss_sched.gmm)`（`stage` 硬编码为 1，其余动态）  
> 调用 `compute_align_loss`

**GMM 更新：** 正常保留（每 10 epoch 调用 `collect_cmss_values` + `cmss_sched.update_gmm`）

**损失权重：** 固定为 0.5/0.5（不调用 `cmss_sched.get_loss_weights`）

**阶段切换：** 不切换，始终 stage=1（即使调用 `cmss_sched.get_stage(epoch)` 返回 0 或 2，也忽略）

#### 运行命令

```bash
python -m src.train_csma \
    --dataset flir_v1 \
    --data-root FLIR_License/train \
    --out-dir outputs_abl_c3_gmm_single_b \
    --cmss-ablation-mode gmm_single_b \
    --epochs 100
```

## 3. 实施顺序与风险提示

### 3.1 推荐的执行顺序

```
Step 1  实现 config.py 的 cmss_ablation_mode 字段（改动最小，所有 C 系列共用）

Step 2  实现 train_csma.py 的分支逻辑（C-1/C-2/C-3 三个 if-elif 分支）
        先用 smoke test（--max-steps 20 --epochs 2）验证三个分支都能正常跑通

Step 3  并行训练 C-1 / C-2 / C-3（各 100 epoch，如果有多卡可同时跑）
        C-4 主实验已有结果

Step 4  评估 C-1/C-2/C-3，填入消融表

Step 5  实现 compute_mean_proto.py（预计算均值原型）
        实现 csma_variants.py（CSMAMeanProto 类）
        实现 train_csma.py 对 --variant mean_proto 的支持
        
Step 6  训练 B-2（均值原型），与主实验（B-3）对比

Step 7  讨论 B-1（双流实时）是否需要实现
```

### 3.2 Smoke Test 验证（实现后必做）

每个新增的消融分支实现后，都先用以下命令验证 2 epoch 不报错：

```bash
python -m src.train_csma \
    --dataset flir_v1 \
    --data-root FLIR_License/train \
    --out-dir outputs_smoke_test \
    --cmss-ablation-mode <模式> \
    --epochs 2 \
    --max-steps 20
```

重点检查：
- 损失不为 NaN
- `mask` 的形状和 `feat_ir` / `feat_rgb` 的 shape 一致（`[B, L]`）
- 对于 C-1（随机掩码），`feat_ir.shape[1]` 需要和生成的随机 mask `L` 维度一致（DINO encoder 输出的 `L_total` 而非 IRE 的 `H/8 × W/8`，注意维度来自 hook 捕获的特征）
- C-2/C-3 调用 `build_cmss_mask` 时 `gmm=None` 或 `gmm=cmss_sched.gmm` 参数传对

### 3.3 关键维度注意事项

L_align 里的 `feat_ir` 和 `feat_rgb` 都来自 **DINO encoder 的 hook 输出**，shape 是 `[B, L_total, 256]`。`L_total` 是 DINO 多尺度特征展平后的总 patch 数，和 IRE 的 `H/8 × W/8` 不同（前者是多尺度拼接，后者是单尺度）。

随机掩码（C-1）生成时，`L` 必须用 `feat_ir.shape[1]`（动态获取），**不能**硬编码为 `(img_size//8)**2`。

### 3.4 B-1（双流实时）的决策

根据前面的分析，推荐方案如下：

> **先不实现 B-1 的训练代码**，在论文表格的该行注释：  
> *"双流实时变体推理时无法获得 RGB 图像，引用 M-SpecGene (ICCV'25) 在 FLIR 上的公开结果作为参考上界。"*

若审稿阶段被 R3 质疑，再补充实现（训练时用实时 RGB 特征做 K/V，推理时 fallback 到均值原型，本质上退化为 B-2）。这样可以节约约 1 个 GPU×天的计算资源，优先保证 B-2/C-1/C-2/C-3 的结果完整。
